5 # NAME
6 
7 xdrc - RFC4506 XDR compiler for libxdrpp
8 
9 # SYNOPSIS
10 
11 xdrc {-hh|-serverhh|-servercc} [-o _outfile_] [-DMACRO=val...] _input_.x
12 
13 # DESCRIPTION
14 
15 `xdrc` compiles an RFC4506 XDR (external data representation) file
16 into a C++11 header file, creating a new C++ type for each type
17 defined in XDR.
18 
19 ## Native representations
20 
21 `xdrc` uses the following representations for XDR types in C++:
22 
23 * XDR structs are translated field-for-field into C++ structures.
24 
25 * XDR unions are translated into a structure in which each field is
26  actually a method that either returns a reference to the underlying
27  value, or throws an exception if that field is not currently
28  selected. The discriminant method with no arguments returns the
29  value, and with a value sets the discriminant (also constructing the
30  appropriate union field). For example, if the XDR file contains:
31 
32  union MyType switch (unsigned discriminant) {
33  case 0:
34  int zero;
35  case 1:
36  string one<>;
37  default:
38  void;
39  }
40 
41  The C++ type can be accessed as follows:
42 
43  MyType mt;
44  mt.discriminant(1);
45  mt.one() = "hello";
46  std::cout << mt.one() << std::endl;
47  mt.discriminant(0);
48  mt.zero() = 5;
49 
50  std::cout << mt.one() << std::endl; // throws xdr_wrong_union
51 
52 * XDR `bool` is represented as a C++ `bool` (with XDR's `TRUE`
53  translated to `true` and `FALSE` to `false`).
54 
55 * XDR [unsigned] `int` and `hyper` types are represented as the
56  cstdint types `std::int32_t`, `std::uint32_t`, `std::int64_t`, and
57  `std::uint64_t`. As per RFC4506, no types narrower than 32 bits
58  exist.
59 
60 * XDR enums are translated into simple (non-class) enums with
61  underlying type `std::int32_t`.
62 
63 * XDR pointers (`*`) are translated into C++ `xdr:pointer`, a subtype
64  of `std::unique_ptr`. `xdr::pointer` adds an `activate()` method
65  that allocates an object of the appropriate type (if the pointer is
66  null) and returns a reference to the current object.
67 
68 * XDR fixed-length arrays are translated into C++ `xdr::xarray`, a
69  subtype of `std::array`.
70 
71 * XDR variable-length arrays (`type field<N>`) are translated into C++
72  `xdr::xvector<T,N>`, a subtype of `std::vector<T>`, where `N`
73  represents the maximum size. Static constexpr method `max_size()`
74  returns the maximum size.
75 
76 * XDR opaque is translated into C++ `std::uint8_t`, but as per
77  RFC4506, opaque may only appear as part of a fixed- or
78  variable-length array declaration.
79 
80 * XDR `string<N>` is translated into `xdr::xstring<N>`, a subtype of
81  string encoding the maximum size. Static constexpr method
82  `max_size()` returns the maximum size.
83 
84 ## Extensions to RFC4506
85 
86 `xdrc` supports the following extensions to the syntax defined in
87 RFC4506:
88 
89 * The source file is run through the C preprocessor. Macro XDRC is
90  pre-defined to 1, permitting the use of #ifdef to take advantage of
91  other xdrc-specific features. Also, with the `-hh` option,
92  `XDRC_HH=1` is pre-defined, and with the `-serverhh` and `-servercc`
93  options, `XDRC_SERVER=1` is predefined.
94 
95 * Lines beginning with a `%` sign are copied verbatim into the output
96  file.
97 
98 * Portions of the input file may be bracketed by `namespace myns {`
99  ... `}`. The corresponding C++ will be embedded in the same
100  namespace.
101 
102 * Type names may include a namespace scope `::`, so as to be able to
103  make use of types defined in a different namespace. (Alternatively,
104  a literal can be used, such as `%using namespace otherns;`.)
105 
106 The namespace-related extensions should be used sparingly if
107 compatibility with other languages and XDR compilers is desirable.
108 While it may be useful to enclose an entire source file in a
109 `namespace` for consistency, it is a good idea to wrap such directives
110 inside `#if XDRC` / `#endif` conditionals. Heavy use of namespaces
111 for internal structuring purposes (e.g., wrapping every `enum` in its
112 own namespace) is a bad idea as it will make protocols incompatible
113 with RFC4506.

115 ## Serialization and traversing data structures
116 
117 A template class `xdr::xdr_traits<T>` is used to hold metadata for
118 each native C++ type `T` representing an XDR type. For XDR structs,
119 unions, arrays and pointers, this traits class contains two important
120 static methods:
121 
122  template<class Archive> void save(Archive &archive, const T &);
123  template<class Archive> void load(Archive &archive, T &);
124 
125 These methods use `archive` as a function object and call it on every
126 field in the data structure. Hence, the type `Archive` can have an
127 overloaded `operator()` that does different things for different
128 types. To implement an archive, you will need to support the
129 following types:
130 
131 > * `bool`, `std::int32_t`, `std::uint32_t`, `std::int64_t`,
132 > `std::uint64_t`, `float`, `double`, `xdr::xstring`,
133 > `xdr::opaque_array` and `xdr::opaque_vec` (the latter two are not
134 > considered containers, despite being implemented in terms of
135 > `xarray` and `xvector`).
136 
137 > * The `xdr::xarray`, `xdr:xvector`, and `xdr::pointer` containers of
138 > the above types (or their supertypes `std::array`, `std::vector`,
139 > and `std::unique_ptr`).
140 
141 > * Any field types that are themselves XDR structures.
142 
143 For debugging purposes and formats (such as JSON) that need access to
144 field names, it is also possible for the `Archive` type to receive the
145 field names of fields that are traversed. The following template (in
146 the `xdr::` namespace) can be specialized to prepare arguments by
147 bundling them with their names:
148 
149  template<typename Archive> struct archive_adapter {
150  template<typename T> static void
151  apply(Archive &ar, T &&t, const char *) {
152  ar(std::forward<T>(t));
153  }
154  };

156 ## Program and version representations
157 
158 For each `version` block (declared inside a `program` block, as
159 documented in Section 12.2 of RFC5531), `xdrc` generates a C++ struct
160 of the same name. As a consequence, versions should use unique names.
161 The version struct generated by `xdrc` contains no data fields, but
162 rather is used to encode information about all procedures in the
163 interface for use by templates. Specifically, the version struct
164 contains the following fields:
165 
166 * `program` - A `uint32_t` encoding the program number specified in
167  the XDR input file.
168 
169 * `version` - A `uint32_t` encoding the program number specified in
170  the XDR input file.
171 
172 * `static constexpr const char *program_name()` - A static method
173  returning the textual name of the program.
174 
175 * `static constexpr const char *version_name()` - A static method
176  returning the textual name of the version.
177 
178 * For each procedure `myproc`, an inner struct `myproc_t` encoding the
179  following metadata about the procedure:
180 
181  * `interface_type` - a typedef of the version struct containing
182  this proc struct.
183 
184  * `proc` - a `uint32_t` containing the procedure number.
185 
186  * `proc_name()` - a static constexpr function returning the name
187  of the procedure.
188 
189  * `arg_type` - exists only if the procedure takes 0 or 1 argument.
190  If it takes 0 arguments, then a typedef of void. If it takes
191  one argument, then a typedef of the argument type.
192 
193  * `arg_tuple_type` - a `std::tuple` parameterized by the types of
194  the argument list of this procedure. E.g., for a declaration
195 
196  void myproc(void) = 1;
197 
198  `arg_tuple_type` is `std::tuple<>`. For a declaration
199 
200  void myproc(int, bool) = 1;
201 
202  `arg_tuple_type` is `std::tuple<std::int32_t, bool>`.
203 
204  * `res_type` - the type returned by the procedure, including
205  `void` for procedures declared to return void.
206 
207  * `res_wire_type` - the type returned by the procedure, except for
208  procedures returning `void`, for which `res_wire_type` is
209  typedefed to `xdr_void` (a.k.a., `std::tuple<>`), an empty data
210  structure whose wire representation is 0 bytes.
211 
212  * `dispatch(c, a1, a2, ...)` - a static method that calls
213  `c.myproc(a1, a2, ...)` (where `myproc` is the name of this
214  procedure) and returns the result. Hence, given a procedure
215  metadata structure `P`, a template can use `P::dispatch` to call
216  a method called `myproc` on an arbitrary class without needing
217  to know that the name of the procedure is `myproc`.
218 
219 * `call_dispatch(t, procno, a1, a2, ...)` - calls the template method
220  `dispatch` on object `t`, passing as a template type argument the
221  procedure metadata type corresponding to procedure number `procno`
222  (a `std::uint32_t`). Returns true if `procno` was valid for this
223  program/version. `call_dispatch` is perhaps best illustrated by
224  example. Given the source:
225 
226  program myprog {
227  version myprog1 {
228  void null(void) = 0;
229  void non_null(int) = 1;
230  } = 1;
231  } = 0x40000000;
232 
233  You will get:
234 
235  struct myprog1 {
236  // ...
237  struct null_t { /* ... */ };
238  struct non_null_t { /* ... */ };
239 
240  template<typename T, typename...A> static bool
241  call_dispatch(T &&t, std::uint32_t proc, A &&...a) {
242  switch(proc) {
243  case 0:
244  t.template dispatch<null_t>(std::forward<A>(a)...);
245  return true;
246  case 1:
247  t.template dispatch<non_null_t>(std::forward<A>(a)...);
248  return true;
249  }
250  return false;
251  }
252  };
253 
254 * `_xdr_client` - a template struct, `template<typename T> struct
255  _xdr_client`, containing a `T` (a pointer-like type), and whose
256  constructor arguments are passed to `T`. In addition, this
257  structure contains one method for each procedure, that calls
258  template method `invoke` on the object pointed to by `T` with the
259  following template typename arguments: The first argument is the
260  procedure metadata type for this procedure. The remaining
261  (variadic) arguments are the types of all the arguments passed to
262  this procedure. Continuing the example above, `myprog1` would
263  contain the following structure:
264 
265  template<typename T> struct _xdr_client {
266  T t_;
267  template<typename...ARGS> _xdr_client(ARGS &&...args)
268  : t_(std::forward<ARGS>(args)...) {}
269 
270  template<typename...ARGS> auto
271  null(ARGS &&...args) ->
272  decltype(t_->template invoke<null_t>(
273  std::forward<ARGS>(args)...)) {
274  return t_->template invoke<null_t>(
275  std::forward<ARGS>(args)...);
276  }
277 
278  template<typename...ARGS> auto
279  non_null(ARGS &&...args) ->
280  decltype(t_->template invoke<non_null_t, int>(
281  std::forward<ARGS>(args)...)) {
282  return t_->template invoke<non_null_t, int>(
283  std::forward<ARGS>(args)...);
284  }
285  };

287 # OPTIONS
288 
289 \-hh
290 : Selects C++ header file output. This is the main output format,
291  and its output is required for use with libxdrpp.
292 
293 \-serverhh
294 : Generates a C++ header file containing declarations of objects you
295  can use to implement a server for each interface, using
296  `srpc_tcp_listener` or `arpc_tcp_listener`. See the EXAMPLES
297  section.
298 
299 \-servercc
300 : Generates a .cc file containing empty method definitions
301  corresponding to the object files created with `-serverhh`.
302 
303 \-a, -async
304 : With `-serverhh` or `-servercc`, says to generate scaffolding for
305  an event-driven interface to be used with `arpc_tcp_listener`, as
306  opposed to the default `srpc_tcp_listener`.
307 
308 \-p, -ptr
309 : With `-serverhh` or `-servercc`, says to generate methods that take
310  arguments and return values as `unique_ptr<T>`. The default is to
311  pass arguments by C++ reference. Note that the library works with
312  both references and `unique_ptr` arguments, so this argument only
313  says what one would like to start out with, and one can later edit
314  individual prototypes to change pointers to references.
315 
316 \-o _outfile_
317 : Specifies the output file into which to write the generated code.
318  The default, for `-hh`, is to replace `.x` with `.hh` at the end
319  of the input file name. `-serverhh` and `-servercc` append
320  `.server.hh` and `.server.cc`, respectively. The special
321  _outfile_ `-` sends output to standard output.
322 
323 \-DMACRO=val
324 : The input file is run through the C preprocessor, and this option
325  adds additional defines. (Note that the symbol `XDRC` is always
326  defined to 1, if you wish to test for xdrc vs. other RPC
327  compilers.)

329 # EXAMPLES
330 
331 Consider the following XDR program definition in a file myprog.x:
332 
333  typedef string big_string<>;
334 
335  program MyProg {
336  version MyProg1 {
337  void null(void) = 0;
338  big_string hello(int) = 1;
339  big_string goodbye(big_string) = 2;
340  } = 1;
341  } = 0x2dee1645;
342 
343 The `-serverhh` option will generate a header with the following
344 class:
345 
346  class MyProg1_server {
347  public:
348  using rpc_interface_type = MyProg1;
349 
350  void null();
351 
352  unique_ptr<big_string>
353  hello(unique_ptr<int> arg);
354 
355  unique_ptr<big_string>
356  goodbye(unique_ptr<big_string> arg);
357  };
358 
359 You have to add any fields you need to this structure, then
360 implement the three methods corresponding to the interface. (Note
361 the very important type `rpc_interface_type` tells the library
362 which interface this object implements.) Given such an object,
363 you can then implement a TCP RPC server (that registers its TCP
364 port with rpcbind) as follows:
365 
366  #include <xdrpp/server.h>
367  #include "xdrpp/myprog.server.h"
368 
369  using namespace xdr;
370 
371  int
372  main(int argc, char **argv)
373  {
374  MyProg1_server s;
375  srpc_tcp_listener rl;
376  rl.register_service(s);
377  rl.run();
378  return 1;
379  }
380 
381 To implement a simple client that talks to this server, you can use
382 code like the following:
383 
384  #include <iostream>
385  #include <xdrpp/srpc.h>
386  #include "myprog.h"
387 
388  using namespace std;
389  using namespace xdr;
390 
391  int
392  main(int argc, char **argv)
393  {
394  unique_fd fd = tcp_connect_rpc(argc > 2 ? argv[2] : nullptr,
395  MyProg1::program,
396  MyProg1::version);
397  srpc_client<MyProg1> c{fd.get()};
398  unique_ptr<big_string> result = c.hello(5);
399  cout << "The result of hello(5) is " << *result << endl;
400  return 0;
401  }

403 ## Compilation
404 
405 The generated `xdrc` output files must be compiled with a compiler
406 supporting C++11 (or later, such as C++14) and expect to be used with
407 libxdrpp. This typically requires extra compiler flags (e.g.,
408 `-std=c++11`). You can use pkgconfig to find the location of the
409 headers and libraries. With CXX set to both g++ and clang++, the
410 following Makefile variables work:
411 
412  CXXFLAGS = -std=c++11 `pkg-config --cflags xdrpp`
413  LIBS = `pkg-config --libs xdrpp`

430 # BUGS
431 
432 Certain names that are legal in XDR cannot be used as type or field
433 names. For example, C++ keywords are not allowed (`namespace`,
434 `template`, etc.). In addition, `xdrc` uses a number of names
435 beginning with underscores (especially names beginning with prefix
436 `_xdr_`). Hence you should avoid starting your field names with
437 underscore. Union types use private fields that have the names of the
438 XDR fields with underscore appended. Hence, in a union you cannot use
439 two field names one of which is the other with an underscore appended.
440 
441 `xdrc` translates an XDR `quadruple` to C++ type called `quadruple`,
442 but most compilers do not have such a type. Moreover, libxdrpp does
443 nothing to support such a type.
444 
445 IEEE 754 floating point allows for many different NaN (not a number)
446 values. During serialization, xdrpp simply passes the bytes of a
447 floating point value through as-is (byteswapping on little-endian
448 machines). Different C++ compilers and libraries could conceivably
449 produce different NaN values from the same code. Hence, in the
450 presence of floating point, the serialization output of seemingly
451 deterministic code may depend on the compiler.
